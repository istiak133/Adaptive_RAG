================================================================================
PART 3: ADVANCED DATABASE SCHEMA & RELATIONAL DESIGN
================================================================================
This document details the best-practice relational database design for the
Knowledge Base, solving the challenge of cross-section topic tracking.
================================================================================

1. THE CROSS-SECTION TOPIC PROBLEM
----------------------------------
Concepts like "Echo Lock" appear in multiple places in the SLATEFALL dossier:
- Section 2 (§2.13) -> Core definition and parameters.
- Section 5 (§5.3)  -> Tactical usage in combat.
- Section 9 (§9.4)  -> A case study incident.
- Section 10 (§10.1)-> Glossary reference.

If the user fails an Echo Lock question in Section 2, the system must remember this weakness when they study Section 9, but adjust the questions based on whether it is a primary topic (Section 2) or a mention (Section 9).

2. ADVANCED SQL SCHEMA DESIGN
------------------------------
To solve this, we make "Topics" a first-class entity and design a normalized schema:

A. sections
   Stores the 10 main section headers.
   - id (INT, PK)
   - title (TEXT)
   - raw_text_length (INT)
   - chunk_count (INT)

B. topics
   Stores all unique concepts extracted from the PDF.
   - id (INT, PK)
   - name (TEXT, e.g., "Echo Lock")
   - slug (TEXT, UNIQUE, e.g., "echo_lock" to unify cases like "Echo-Lock")
   - category (TEXT, e.g., "power_mechanic", "adversary", "event")
   - importance (TEXT, e.g., "critical", "normal", "minor")

C. section_topics
   Many-to-many table mapping which topics appear in which sections, and their depth.
   - section_id (INT, FK)
   - topic_id (INT, FK)
   - depth (TEXT, 'primary' | 'secondary' | 'mention')
   - relevance_score (REAL)
   * Primary: Full definition (e.g., Echo Lock in Sec 2).
   * Secondary: Discussed at length (e.g., Echo Lock in Sec 5).
   * Mention: Briefly referenced (e.g., Echo Lock in Sec 9).

D. chunks
   Text fragments of the PDF.
   - id (INT, PK)
   - section_id (INT, FK)
   - sub_section_id (TEXT, e.g., "2.13")
   - sub_section_title (TEXT)
   - content (TEXT)
   - token_count (INT)

E. chunk_topics
   Many-to-many table mapping chunks to the topics they cover.
   - chunk_id (INT, FK)
   - topic_id (INT, FK)

F. sessions
   Logs prep sessions.
   - id (INT, PK)
   - sections_studied (TEXT, JSON array of section IDs)
   - total_questions (INT)
   - correct_count (INT)
   - wrong_count (INT)
   - score_pct (REAL)
   - difficulty_level (TEXT)
   - adaptive_context (TEXT, JSON representation of LLM history sent)
   - started_at, completed_at (TIMESTAMP)

G. questions
   Stores generated MCQs.
   - id (INT, PK)
   - session_id (INT, FK)
   - section_id (INT, FK)
   - source_chunk_ids (TEXT, JSON array of chunks used)
   - question_text, choice_a, choice_b, choice_c, choice_d (TEXT)
   - correct_answer (TEXT, 'A'|'B'|'C'|'D')
   - explanation (TEXT)
   - source_quote (TEXT, exact quote verified from PDF)

H. question_topics
   Maps which topics are tested by a question. A question can test multiple topics!
   - question_id (INT, FK)
   - topic_id (INT, FK)
   - is_primary (BOOLEAN, indicates if this topic is the main focus of the question)

I. answers
   Logs user answers.
   - id (INT, PK)
   - question_id (INT, FK, UNIQUE)
   - user_answer (TEXT)
   - is_correct (BOOLEAN)

J. topic_mastery (Global)
   Tracks user performance on a topic globally across the whole PDF.
   - topic_id (INT, FK, UNIQUE)
   - times_asked, times_correct, times_wrong (INT)
   - current_streak (INT, positive for correct, negative for wrong)
   - weight (REAL, range 0.1 to 5.0, default 1.0)
   - is_mastered (BOOLEAN)

K. section_topic_mastery (Section-specific)
   Tracks performance on a topic specific to a section (e.g., user knows the definition of Echo Lock in Sec 2, but fails the practical application in Sec 9).
   - section_id (INT, FK)
   - topic_id (INT, FK)
   - times_asked, times_correct, times_wrong (INT)
   - weight (REAL)
   * Primary key: (section_id, topic_id)

3. THE ADAPTIVE WEIGHT UPDATE LOGIC
-----------------------------------
When the user answers a question, we update both global and section-specific mastery:
- If correct:
  * Increase times_correct.
  * Streak increases (current_streak = max(streak, 0) + 1).
  * Recalculate weight down. If streak >= 3, set is_mastered = True and weight = 0.1.
- If wrong:
  * Increase times_wrong.
  * Streak decreases (current_streak = min(streak, 0) - 1).
  * Recalculate weight up.
  * If the topic was mastered before, mark is_mastered = False and boost weight back to 2.5 (Regression Detection).
- Impact Factor: If the topic was only a secondary topic in the question, the weight update is scaled by 0.5.

Weight Formula:
- Base Weight = 1.0 + (times_wrong * 0.5 * impact)
- If streak < 0 (consecutive wrong): multiply by (1.0 + abs(streak) * 0.3)
- If streak >= 2 (consecutive correct): multiply by 0.3
- Cap weight between 0.1 and 5.0.

4. QUESTION ALLOCATION LOGIC (WEIGHT -> QUESTION COUNT)
-------------------------------------------------------
When studying a section:
1. Fetch all topics mapped to that section from `section_topics`.
2. Compute the "effective weight" for each topic:
   - If is_mastered = True: effective_weight = 0.1.
   - Else: effective_weight = global_weight * depth_multiplier.
     * depth_multiplier: primary = 1.0, secondary = 0.6, mention = 0.3.
3. Normalize the effective weights and distribute the N questions (default 5) proportionally.
4. If a topic has a weight > 2.0 (high priority weakness), force it to receive at least 1 question.
5. Allocate remaining slots to unasked/new topics to ensure exploration.
