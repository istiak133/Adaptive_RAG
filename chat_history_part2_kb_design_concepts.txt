================================================================================
PART 2: KNOWLEDGE BASE (KB) DESIGN & HIGH-LEVEL STRATEGY
================================================================================
This document captures our discussion on what a Knowledge Base (KB) is, how it
operates, and the hybrid database strategy we chose for this project.
================================================================================

1. WHAT IS THE KNOWLEDGE BASE (KB)?
-----------------------------------
- Think of the KB as the system's "memory" or "notebook."
- If a teacher gives a student an exam, they note down which topics the student failed. The next day, the teacher creates a test focusing heavily on those weak topics.
- In this project, the KB stores:
  * The session details (which sections were studied, when, and the score).
  * The questions generated by the LLM (question text, choices, correct answer, explanation, topic name).
  * The user's responses (which option they chose and whether it was correct).
  * Recalculated parameters (topic weights, streaks, mastery status).

2. THE ADAPTIVE LOOP: HOW KB INTERFACES WITH LLM
------------------------------------------------
Every session follows this loop:
1. User enters: "I want to study Section 8."
2. System queries the KB:
   - "Have we studied Section 8 before?" -> Yes (Iteration 1).
   - "What topics in Section 8 did the user fail?" -> "Combat Doctrine" and "Echo Lock".
   - "What questions were already asked?" -> [List of past question texts].
3. System builds a smart prompt for the LLM:
   - "Generate 5 MCQs from Section 8.
      PRIORITIZE these weak topics: Combat Doctrine, Echo Lock.
      AVOID repeating these questions: [past questions].
      Here is the text chunk for Section 8..."
4. LLM generates targeted, non-repetitive MCQs.
5. User answers, system scores, and writes the results back to the KB, updating the weights for the topics.

3. HYBRID STORAGE APPROACH
--------------------------
To build the "best way" rather than the "simplest way," we decided on a hybrid database architecture:

A. Relational Database (SQLite)
   - Handles structured data perfectly.
   - Ideal for tracking metrics: session logs, question details, exact answers, topic weights, streaks, and timestamps.
   - Allows us to run fast, structured SQL queries like: "Get the top-3 weakest topics in Section 8."

B. Vector Database (ChromaDB)
   - Handles unstructured semantic data.
   - We use ChromaDB for two things:
     1. Semantic chunking: Ingesting the PDF and storing text chunks with vector embeddings. When a user is weak in a topic, we use semantic search to fetch the most relevant text chunks containing that topic to feed to the LLM.
     2. Question Deduplication: Even if we tell the LLM "do not repeat these exact questions," it might write a slightly different question testing the exact same thing (semantic duplicate). Before presenting questions, we embed the new questions and query ChromaDB to check if they are too semantically similar (>85% match) to past questions. If so, we reject and regenerate.

4. THE KB SNAPSHOT REQUIREMENT
------------------------------
- The assessment brief requires a "human-readable snapshot" after each iteration (e.g., kb_snapshot_iter1.json).
- This is a JSON export of the current state of the KB.
- It includes:
  * The last session details.
  * The history of the last 5 sessions.
  * The current state of topic performance (weights, correctness rate, streaks, mastered status) across all sections.
- This allows a reviewer to audit the system's memory and verify that weights are changing correctly.
